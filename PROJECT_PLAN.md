## 1. Task Overview

This TerminalBench Edition 2 (non-milestone) task asks the agent to build a **C++ recovery pipeline** that ingests corrupted mTLS service-trust graphs encoded as GraphML (XML), repairs structural and cryptographic defects, validates the result against the official GraphML schema, verifies trust relationships through a provided Flask verifier API, and loads only verified records into PostgreSQL using `libpq` bulk `COPY`.

The task sits at the intersection of **security/cryptography**, **API integration**, and **tool-specific systems work** (C++, PostgreSQL, XML schema validation, Ed25519 signature verification).

**Harness task name:** `snorkel/recover-mtls-trust-graphs-postgresql`  
**Difficulty:** Hard | **Domain:** Security & Cryptography | **Subtypes:** API Integration, Tool Specific

---

## 2. Problem Statement

An operations team exported mTLS service-trust graphs as GraphML documents. During transfer and archival, the artifacts became **corrupted** and **incomplete**:

- XML records were **split** across `head.graphml` and `tail.graphml` fragments.
- **Key declarations** required for GraphML data attributes were missing.
- **Duplicate service nodes** appeared with conflicting `version` attributes.
- **Ed25519 signatures** on authorization edges were **detached** to `detached_signatures.json`.

Trust cannot be assumed from structure alone. Each service identity must be checked against issuer keys, revocation status, and signed trust-edge validity. Only after repair, schema validation, and API verification should data enter PostgreSQL.

---

## 3. Objective

Build a complete, runnable recovery system that:

1. **Repairs** corrupted GraphML input into a well-formed, deduplicated graph.
2. **Fetches** the live GraphML XSD from `http://graphml.graphdrawing.org/xmlns/1.0/graphml.xsd` at runtime and **validates** repaired XML (schema is **not** vendored).
3. **Verifies** nodes and edges via the Flask verifier API on port **5001**.
4. **Ingests** only verified records into PostgreSQL using **`libpq` `COPY FROM STDIN`**.
5. **Emits** a schema-validated JSON recovery report.

**Resolved design decisions:**

| Decision | Resolution |
|----------|------------|
| Duplicate merge policy | Keep node with higher `version`; on tie, keep later document order |
| Partial success | `status: "partial"` when XSD passes but some nodes/edges rejected |
| Canonical edge payload | `{source}\|{target}\|{policy}` UTF-8, no trailing newline |
| Report schema | `environment/app/schema/report.schema.json` |
| XSD URL | HTTP (HTTPS cert hostname mismatch on official host) |

## 10. Expected Workflow

```
Corrupted fragments → Repair (merge, keys, dedupe, reattach sigs)
                   → XSD validate (live fetch)
                   → Flask verify (key, revocation, edge)
                   → libpq COPY (verified rows only)
                   → JSON report
```

### Step-by-step

1. **Bootstrap** — `start_services.sh` → PostgreSQL + Flask on 5001.
2. **Repair** — Merge fragments; inject `<key>` before `<graph>`; dedupe by `service_id`/`version`; reattach `d_sig`.
3. **Fetch XSD** — HTTP GET from graphml.graphdrawing.org.
4. **Validate** — libxml2 XSD check on `repaired.graphml`.
5. **Verify nodes** — `/verify/key` + `/verify/revocation`; skip revoked/invalid.
6. **Verify edges** — `/verify/edge` with canonical `{source}|{target}|{policy}` payload.
7. **Ingest** — `COPY FROM STDIN` into `services` and `trust_edges`.
8. **Report** — Write `report.json`; exit 0 on `success` or `partial`.
